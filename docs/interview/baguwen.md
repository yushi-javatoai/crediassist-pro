## 一、多Agent架构基础 (8题)

### Q1: 单Agent vs 多Agent，什么时候用多Agent？

**答**：

单Agent适合任务单一、工具集小、不需要专业化分工的场景（如简单QA机器人）。多Agent适合以下场景：

1. **任务复杂度高**：需要拆解为多个子任务，各自需要不同的Prompt和工具集
2. **专业化分工**：不同子任务需要不同的"专家"——如金融合规和产品咨询需要完全不同的知识和判断标准
3. **上下文隔离**：单Agent把所有Prompt和工具塞进一个上下文窗口，Token消耗大且容易混淆
4. **并行处理**：多个子任务可以并发执行，降低总延迟

**追问：多Agent的缺点是什么？**

- 系统复杂度增加：编排、通信、状态同步都需要额外设计
- 延迟增加：Agent间的路由和消息传递有额外开销
- 调试困难：一个请求可能经过3-4个Agent，问题定位需要全链路追踪
- 成本可能更高：多次LLM调用（意图分类一次、业务处理一次、合规审查一次）

---

### Q2: Supervisor模式 vs Peer-to-Peer模式的优劣？

**答**：

| 维度 | Supervisor模式 | Peer-to-Peer模式 |
|------|---------------|-----------------|
| 控制方式 | 中央协调者分发和汇总 | Agent间直接通信 |
| 适合场景 | 需要统一审查的场景（客服、金融） | 需要协商决策的场景（研究、辩论） |
| 优点 | 流程可控、容易加审查环节 | 灵活、去中心化、无单点故障 |
| 缺点 | Supervisor是单点、所有请求都经过它 | 难以保证一致性、调试复杂 |
| 状态管理 | 集中式State，由Supervisor维护 | 分布式State，各Agent各自维护 |

在智能客服场景选择Supervisor的原因：**所有回复必须经过合规审查**，这天然需要一个中央节点来保证。

**追问：Supervisor成为性能瓶颈怎么办？**

1. Supervisor只做路由决策（轻量级），不做业务处理
2. 对Supervisor的LLM调用做缓存——相似问题的路由决策可以cache
3. 多实例部署Supervisor，按session_id做一致性哈希分流

### Q3: 解释一下ReAct框架的原理？

**答**：

ReAct = **Re**asoning + **Act**ing。核心思想是让LLM在推理和行动之间交替执行：

```
循环：
  1. Thought: LLM思考当前状况，决定下一步
  2. Action: 调用一个工具（搜索、查询、计算等）
  3. Observation: 获取工具返回的结果
  4. 回到步骤1，基于新Observation继续推理
  直到：LLM判断可以给出最终答案
```

与纯Chain-of-Thought的区别：CoT只有推理没有行动，ReAct可以在推理过程中调用外部工具获取实时信息。

**追问：ReAct有什么问题？怎么改进？**

- **死循环**：Agent可能在Thought和Action之间无限循环。解决：设最大循环次数 + 超时熔断
- **工具选择错误**：LLM可能选错工具。解决：优化工具描述、Few-shot示例
- **效率低**：每次循环都要调用LLM。解决：Plan-then-Execute模式，先一次性规划所有步骤

### Q4: 多Agent间怎么通信？有哪些通信模式？

**答**：

1. **共享状态（Shared State）**：所有Agent读写同一个State对象。LangGraph的主要模式。优点是简单直接，缺点是需要处理并发写入冲突。

2. **消息传递（Message Passing）**：Agent之间通过消息队列通信。适合分布式部署。如Kafka、RabbitMQ。

3. **工具调用（Agent-as-Tool）**：一个Agent把另一个Agent当作工具来调用。AWS Agent Squad的SupervisorAgent就是这种模式。

4. **黑板模式（Blackboard）**：所有Agent向一个共享"黑板"写入信息，其他Agent可以读取。适合需要多Agent贡献知识的场景。

我们的项目采用**共享状态模式**，原因是：
- 所有Agent在同一个进程内运行，无需网络通信
- LangGraph的StateGraph天然支持这种模式
- Supervisor可以直接读取所有子Agent的结果进行汇总

---

### Q5: 什么是Human-in-the-Loop？在Agent系统中怎么实现？

**答**：

Human-in-the-Loop（HITL）指在Agent自动化流程中插入人工审核/决策节点。

实现方式：
1. **LangGraph Breakpoint**：在Graph的特定节点设置interrupt，执行到该节点时暂停，等待人工输入后继续
2. **Checkpoint + 恢复**：Agent执行到断点时，将完整State持久化（checkpoint），人工审核后恢复执行
3. **审批队列**：敏感操作（如大额退款）发送到审批队列，人工审批后回调Agent继续

在我们的项目中，当合规审查Agent检测到高风险内容时，会自动创建工单并暂停自动回复，等待人工审核。

### Q6: Agent的幻觉问题怎么解决？

**答**：

Agent幻觉指LLM生成与事实不符的内容。在客服场景下特别危险——比如承诺不存在的退款政策。

多层防御策略：
1. **RAG约束**：在Prompt中明确要求"只基于检索到的文档回答，不要编造"，并提供Few-shot反例
2. **合规审查Agent**：专门检测越权承诺和虚假信息
3. **引用追踪**：回答中标注引用来源，方便人工验证
4. **置信度阈值**：当RAG检索相似度低于阈值时，不自动回答，转人工
5. **自我纠正（Self-Correction）**：让Agent对自己的输出进行二次审视，检查是否与文档矛盾

---

### Q7: Token消耗优化有哪些方案？

**答**：

1. **分层记忆**：不是每次都把全部历史塞进prompt。工作记忆只存关键状态，短期记忆做滑动窗口，长期记忆用向量检索按需取用
2. **Prompt压缩**：用LLM对历史对话做摘要，代替完整历史
3. **缓存**：相似问题的路由决策和检索结果做缓存
4. **模型分级**：意图路由用小模型（便宜快速），知识生成用大模型（质量优先）
5. **早退机制**：简单问题（FAQ命中）直接返回，不走完整Agent链路

在我们的项目中，分层记忆使Token消耗降低了40%。

---

### Q8: 多Agent系统的容错和降级策略？

**答**：

1. **超时熔断**：每个Agent设超时上限（如10秒），超时自动降级
2. **重试策略**：对可重试错误（如API限流）做指数退避重试，最多3次
3. **降级回复**：Agent异常时返回兜底话术 + 自动创建工单转人工
4. **隔舱设计**：一个子Agent故障不影响其他Agent执行
5. **Checkpoint恢复**：对长流程做检查点，异常后可从检查点恢复

## 二、RAG知识检索 (8题)

### Q9: 描述一下完整的RAG流程？

**答**：

```
用户问题 → Query改写 → 向量化 → 向量检索(Top-K) → 重排序(Re-rank) → 上下文注入 → LLM生成 → 引用标注
```

1. **Query改写**：将口语化问题转为检索友好的查询（如"怎么退钱"→"退款政策 申请流程 时限"）
2. **向量化**：用Embedding模型将Query转为向量
3. **向量检索**：在向量数据库中做近似最近邻搜索，取Top-K候选
4. **重排序**：用Cross-Encoder或LLM对候选文档按相关性重排
5. **上下文注入**：将排序后的Top文档拼接到Prompt中
6. **LLM生成**：基于上下文生成回答
7. **引用标注**：标注回答引用了哪些文档

---

### Q10: 文档分块（Chunking）有哪些策略？各自优缺点？

**答**：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定长度 | 按字符/Token数切分 | 简单快速 | 可能切断句子语义 |
| 语义分块 | 按段落/句子分割 | 保持语义完整 | 块大小不均匀 |
| 滑动窗口 | 固定长度+重叠 | 避免边界信息丢失 | 有冗余存储 |
| 递归分割 | 先按段落，段落太长则按句子 | 兼顾语义和大小 | 实现复杂 |

我们的项目采用**固定长度(512 tokens) + 重叠窗口(128 tokens)**，同时优先按段落分割。

**追问：重叠窗口的overlap设多大合适？**

一般设chunk_size的10%-25%。太小会丢失边界信息，太大会增加存储和检索成本。我们用128/512=25%。

---

### Q11: 向量数据库选型（FAISS vs Milvus vs Pinecone）？

**答**：

| 维度 | FAISS | Milvus | Pinecone |
|------|-------|--------|----------|
| 部署 | 单机库，需自管理 | 分布式，云原生 | 全托管SaaS |
| 性能 | 最快（毫秒级） | 快（10ms级） | 中等（API延迟） |
| 规模 | 千万级 | 百亿级 | 百亿级 |
| 成本 | 免费 | 中等 | 最高 |
| 运维 | 需自行管理 | 需K8s运维 | 零运维 |

选型建议：
- 开发/测试：FAISS（零依赖，快速验证）
- 小规模生产（<1000万向量）：FAISS也够用
- 大规模生产（>1亿向量）：Milvus
- 不想运维：Pinecone（但注意数据合规）

### Q12: 检索效果不好怎么优化？

**答**：

1. **Query改写**：用LLM改写用户问题，补充专业术语
2. **混合检索**：向量检索 + BM25关键词检索，合并结果
3. **重排序（Re-rank）**：用Cross-Encoder对Top-K结果重排
4. **优化Embedding模型**：用领域微调的Embedding模型
5. **优化分块策略**：调整chunk_size和overlap
6. **多路召回**：同时用不同的检索策略，取并集
7. **反馈学习**：根据用户反馈标注好/坏case，持续优化

---

### Q13: RAG vs Fine-tuning，怎么选？

**答**：

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 实时（更新文档即可） | 需要重新训练 |
| 成本 | 检索+生成开销 | 训练+部署开销 |
| 可解释性 | 高（可引用来源） | 低（黑箱） |
| 幻觉控制 | 更好（有文档约束） | 较差 |
| 适合场景 | 知识频繁更新、需要引用 | 风格/格式固定、领域深度理解 |

客服场景选RAG：产品信息频繁更新，需要引用政策文档，合规要求可追溯。

### Q14: Embedding模型怎么选？

**答**：

主流选择：
- **OpenAI text-embedding-3-large**：通用效果好，1536维
- **BGE系列**：中文效果最好的开源模型
- **Cohere Embed v3**：多语言支持好
- **Sentence-Transformers**：本地部署首选

选型考虑：语言（中文需要中文优化模型）、维度（越高越精确但越慢）、是否支持本地部署（数据安全要求）。

---

### Q15: 什么是Hybrid Search（混合检索）？

**答**：

结合**稠密向量检索**（语义相似度）和**稀疏检索**（BM25关键词匹配）的检索方法。

原理：
- 稠密检索擅长理解语义（"怎么退钱" ≈ "退款流程"）
- 稀疏检索擅长精确匹配（专有名词、订单号）
- 两者结合：先各自检索Top-K，再用RRF（Reciprocal Rank Fusion）合并排序

实际效果通常比纯向量检索提升5-15%。

---

### Q16: 上下文窗口不够用怎么办？

**答**：

1. **压缩上下文**：用LLM对检索结果做摘要，只保留关键信息
2. **分层注入**：最相关的文档完整注入，次相关的只注入摘要
3. **Map-Reduce**：对每个文档分别生成子回答，最后合并
4. **长上下文模型**：使用128K+窗口的模型（如GPT-4o、Claude）
5. **筛选策略**：严格控制注入文档数量，只注入相关性最高的2-3篇

## 三、MCP工具协议 (4题)

### Q17: MCP协议是什么？和传统API调用有什么区别？

**答**：

MCP（Model Context Protocol）是Anthropic开源的标准协议，定义了AI Agent与外部工具/数据源的通信方式。

与传统API调用的区别：

| 维度 | 传统API | MCP |
|------|---------|-----|
| 集成方式 | N个Agent × M个工具 = N×M个适配器 | N+M个标准化连接 |
| 工具发现 | 硬编码 | 动态发现（tools/list） |
| 协议 | 各自定义 | 统一JSON-RPC 2.0 |
| 扩展性 | 加一个工具改N个Agent | 加一个工具所有Agent自动可用 |

类比：MCP之于AI工具，就像USB-C之于硬件设备——统一接口标准。

---

### Q18: MCP协议包含哪些核心方法？

**答**：

1. **tools/list**：工具发现，Agent查询可用工具列表和参数Schema
2. **tools/call**：工具调用，Agent传入工具名和参数，获取结果
3. **resources/list**：资源发现，列出可用的数据资源
4. **resources/read**：资源读取
5. **prompts/list** + **prompts/get**：提示模板管理

传输方式：JSON-RPC 2.0 over stdio / HTTP+SSE

---

### Q19: 在项目中怎么实现MCP？

**答**：

我们实现了一个MCPToolServer类：
1. **工具注册**：通过装饰器/注解注册工具函数，声明name、description、inputSchema
2. **工具发现**：暴露`/api/tools`端点，Agent启动时自动获取可用工具列表
3. **工具调用**：Agent通过`tools/call`方法传入参数，MCPToolServer路由到对应handler执行
4. **日志记录**：每次调用记录工具名、参数、耗时、成功/失败，用于追踪分析

---

### Q20: 工具调用失败怎么处理？

**答**：

1. **重试**：可重试错误（超时、限流）做指数退避重试
2. **降级**：工具不可用时返回降级结果（如缓存的旧数据）
3. **错误传播**：将错误信息传回Agent，让Agent决定下一步（换工具或告知用户）
4. **熔断器**：连续失败超过阈值，自动熔断该工具，避免雪崩

## 四、分层记忆系统 (4题)

### Q21: 分层记忆系统的三层分别是什么？

**答**：

| 层级 | 存储介质 | 生命周期 | 延迟 | 用途 |
|------|---------|---------|------|------|
| 工作记忆 | 进程内存 | 单次请求 | <1ms | Agent当前推理状态 |
| 短期记忆 | Redis | 30分钟TTL | 1-5ms | 最近20轮对话上下文 |
| 长期记忆 | 向量数据库 | 永久 | 10-50ms | 知识库、用户画像、历史工单 |

三层协作：
1. Supervisor先查工作记忆获取当前状态
2. 需要对话上下文时查短期记忆
3. 需要知识或用户画像时查长期记忆
4. 重要中间结果从工作记忆提升到短期记忆（跨轮保留）

---

### Q22: 短期记忆的淘汰策略？

**答**：

1. **滑动窗口**：只保留最近N轮（我们用20轮）
2. **TTL过期**：Redis自动过期（30分钟无活动）
3. **Token预算**：总Token数超过预算时，从最早的消息开始淘汰
4. **重要性权重**：用户首次描述的问题比中间确认消息重要，可优先保留

**追问：缓存击穿/雪崩怎么处理？**

- 击穿：热点session的缓存过期时大量请求打到后端。解决：互斥锁 + 永不过期(后台异步更新)
- 雪崩：大量session同时过期。解决：TTL加随机偏移量（如30min±5min）

---

### Q23: 长期记忆什么时候写入？

**答**：

1. **知识库批量导入**：系统启动时或管理员上传文档时
2. **对话结束摘要**：会话结束后，将对话摘要和关键实体写入长期记忆
3. **工单信息**：工单创建/更新时写入
4. **用户画像更新**：从对话中提取的用户偏好、常见问题模式

**写入策略**：异步写入，不阻塞主流程。使用消息队列缓冲写入请求。

---

### Q24: 记忆系统在分布式环境下怎么保证一致性？

**答**：

- **工作记忆**：进程内，无需分布式一致性，但需要按session_id做请求亲和（sticky session）
- **短期记忆**：Redis单节点或Redis Cluster，依赖Redis自身的一致性保证
- **长期记忆**：向量数据库（Milvus）支持分布式一致性，写入后需要等flush完成才对读可见

关键设计：session_id的一致性路由——同一个session的所有请求路由到同一个实例，避免工作记忆不一致。

---

## 五、全链路追踪 (4题)

### Q25: 为什么Agent系统特别需要全链路追踪？

**答**：

传统微服务的追踪主要看HTTP调用链。Agent系统更复杂：
1. **路由决策不确定**：同一个问题可能路由到不同Agent
2. **LLM输出不确定**：相同输入可能产生不同输出
3. **多Agent串行/并行**：一个请求可能触发3-4个Agent调用
4. **Token消耗不透明**：不追踪就无法知道钱花在哪里

追踪Agent系统需要额外记录：每个Agent的路由决策原因、LLM的输入/输出、Token消耗、工具调用参数和结果。

### Q27: 追踪数据怎么用于优化系统？

**答**：

1. **延迟分析**：哪个Agent最慢？是LLM调用慢还是向量检索慢？
2. **路由准确率**：统计意图路由的准确率，发现路由错误的Pattern
3. **Token消耗分布**：哪个Agent消耗Token最多？是否有优化空间？
4. **错误率监控**：哪个Agent错误率最高？错误原因是什么？
5. **Badcase分析**：用户投诉时可以回溯完整链路

### Q28: LangSmith和OpenTelemetry的关系？

**答**：

- **OpenTelemetry**：通用的分布式追踪标准，适用于任何系统
- **LangSmith**：LangChain团队的LLM专用观测平台，专门为LLM应用优化

LangSmith现在支持OpenTelemetry格式，可以用OTEL标准收集Trace再导入LangSmith做LLM专用分析。

推荐：用OpenTelemetry做基础设施级追踪（延迟、错误率），用LangSmith做LLM级分析（Prompt优化、回答质量评估）。

## 六、系统设计与性能 (4题)

### Q29: 系统QPS能到多少？瓶颈在哪里？

**答**：

瓶颈分析：
- **LLM API调用**：最大瓶颈。GPT-4o约50-100 TPS，每个请求调用2-3次LLM
- **向量检索**：FAISS毫秒级，Milvus 10ms级，不是瓶颈
- **Redis**：10万+ QPS，不是瓶颈

优化方案：
1. 缓存热点问题的路由决策和检索结果
2. 简单问题（FAQ精确命中）直接返回，不走LLM
3. 意图路由用小模型，降低API成本和延迟
4. 请求排队 + 批处理

Python版预估QPS：50-100（受LLM API限制）
Go版预估QPS：200-500（Go本身开销极低，主要还是LLM瓶颈）

### Q30: 如何做Agent系统的压测？

**答**：

1. **Mock LLM**：压测时用固定响应替代真实LLM调用，测试系统本身的吞吐
2. **渐进加压**：从10 QPS逐步加到目标QPS，观察P99延迟变化
3. **场景覆盖**：构造不同意图的测试用例（咨询、投诉、办理），模拟真实分布
4. **长时间压测**：测试内存泄露、连接池耗尽等问题
5. **工具**：Locust（Python）、k6（Go/JS）、JMeter（Java）

关键指标：QPS、P50/P95/P99延迟、错误率、内存/CPU使用率。

### Q31: Agent系统的安全威胁有哪些？

**答**：

1. **Prompt注入**：用户在消息中嵌入恶意指令，试图改变Agent行为
2. **PII泄露**：Agent在回复中暴露其他用户的个人信息
3. **越权操作**：Agent被诱导执行超出权限的工具调用
4. **供应链攻击**：恶意MCP工具
5. **数据投毒**：知识库中被注入虚假信息

防御措施：
- 输入净化 + 输出审查（合规Agent）
- PII检测和脱敏
- 工具调用权限控制
- 知识库内容审核流程

### Q32: 如果让你重新设计这个系统，你会做什么不同？

**答**：

这是一个很好的反思问题。我会做以下改进：

1. **评估先行**：先建立完善的评估体系（Golden Set + 自动评估Pipeline），再迭代优化，而不是凭感觉调优
2. **Streaming优先**：从一开始就设计SSE流式响应，用户体验会好很多
3. **工具编排图可视化**：提供Admin UI让运营人员可以拖拽配置Agent编排流程
4. **混合检索**：一开始就用Hybrid Search（向量+BM25），而不是纯向量检索
5. **A/B测试框架**：内置A/B测试能力，方便对比不同Prompt/模型/策略的效果
